## ⚠️ 必須補充的關鍵資訊

系統化地分析了執行時需要但目前缺乏的資訊，按照優先級分為：
- 🔴 關鍵缺失（執行前必須補充）
- 🟡 重要缺失（影響執行效率）
- 🟢 次要缺失（可以在執行中學習）

**🔴 關鍵缺失（必須補充）:**
1. 驗證單詳細結構（欄位位置、色碼定義）
2. 後台頁面導航和資料提取步驟
3. 資料比對具體規則

**🟡 重要缺失（建議補充）:**
1. 完整驗證案例演示
2. 常見驗證項目標準流程
3. 環境差異說明

**🟢 次要缺失（可逐步完善）:**
1. Redmine/Slack 整合細節
2. 排程表操作說明

在實際使用 SKILL 之前，**必須**補充以下資訊：

### 1. 驗證單詳細結構(欄位對照表)
**為什麼重要**: 沒有這個資訊，Agent 無法準確定位和更新儲存格

**需要提供**:

[驗證單欄位對照表](./sge-qa-verification/docs/sheet-structure.md)

欄位名稱     | Column 位置 | 資料類型 | 色碼規則 | 填寫規則
-----------|------------|---------|----------|--------
確認方法     | B          | 文字     | 黃底表頭 | 後台路徑導引
確認項目     | C          | 文字     | 黃底表頭 | 驗證描述
確認（測試） | D          | 混合     | 白底(待填) / 灰底(跳過) / 紅底(錯誤) | OK或紅底+說明
担当（測試） | E          | 文字     | 白底(待填) / 灰底(跳過) / 紅底(錯誤) | IW
担当者（測試）| F          | 文字     | 白底(待填) / 灰底(跳過) / 紅底(錯誤) | Agent
日付（測試） | G          | 日期     | 白底(待填) / 灰底(跳過) / 紅底(錯誤) | 動作執行日期
確認（正式） | H          | 混合     | 同測試環境規則
...


**具體色碼**:標註色碼定義
- 黃底表頭: #FFFF00 或其他？
- 白底待填: #FFFFFF
- 灰底跳過: #CCCCCC 或其他？
- 紅底錯誤: #FF0000 或其他？


### 2. 後台頁面操作指南
[後台頁面操作指南](./sge-qa-verification/docs/backend-navigation.md)
**為什麼重要**: Agent 需要知道如何在後台系統中定位和提取資料

**需要提供** (針對每種資料類型):
  - 常見驗證項目的標準流程
    - 每種驗證類型的後台操作步驟
    - 截圖關鍵頁面和元素
  - 記錄 HTML 結構（如有可能）

#### 範例
- ##### Banner 驗證
  ```
  驗證單「確認方法」欄內容: "バナー一覧"

  後台操作步驟:
  1. 訪問 https://common-legacy-app-admin.us.ikemen-sengoku.jp/admin/campaign
  2. 在搜尋框輸入活動名稱: [從驗證單獲取]
  3. 點擊對應的 Banner 記錄
  4. 在詳情頁面找到以下欄位：
    - Banner ID: <xpath 或 CSS selector>
    - 顯示期間: <xpath 或 CSS selector>
    - 圖片 URL: <xpath 或 CSS selector>
    - 連結設定: <xpath 或 CSS selector>
  ```

- ##### Card/Still 驗證
  ```
  驗證單「確認方法」欄內容: "カード発送"

  後台操作步驟:
  1. 訪問 https://common-legacy-app-admin.us.ikemen-sengoku.jp/admin/still
  2. 在 Card ID 輸入框輸入: [從驗證單獲取]
  3. 提取以下資訊：
    - Card Name: <定位方式>
    - Rarity: <定位方式>
    - Character: <定位方式>
  ```

### 3. 資料比對規則細節
**為什麼重要**: 決定什麼情況算「符合」或「不符」

**需要明確**:
- 日期格式的容錯規則（2024/02/01 vs 2024-02-01）
- 數字的比對方式（100 vs 100.0）
- 文字的大小寫敏感度
- 日英文混合時的處理
- 特殊符號的等價性（～ vs -）
